# 「跑一次就好」是怎麼做出來的

一個公家機關辦事指南 PWA 的完整拆解：從一次貸款卡關的親身經驗，到一個免費、零後端、可離線使用的網站。線上版：https://anikievil.github.io/cateye/

## 一、核心概念

### 起點是一張找不到的權狀

這個專案不是「我想做個政府服務網站」的規劃結果，是被逼出來的。

向農會辦房屋土地抵押貸款：財力證明印了、一年份存摺補登了、聯徵費我跟我媽兩人繳了 600 元、核貸也下來了——150 萬、3.68%、7 年、月繳約 20,020。結果送件後才被通知要補「土地使用分區證明」（公所辦、等一週）；補齊送件、到了對保當天，才發現所有權狀正本找不到。權狀遺失只能到地政事務所辦「書狀補給」，法定公告 30 天才能領新權狀，貸款整案直接延後一個多月。

更荒謬的是後面那一段：補發權狀還要先到戶政辦印鑑證明，戶政櫃檯問「要幾份？用途是什麼？」，答不出來，回去問地政，地政也不確定，在兩個單位之間跑了三趟，最後擠出來的答案只有一行字：一份，用途填「權狀申請」。

這件事後來變成 app 裡第一個「情境包」（`loan-mortgage`）——不是抽象案例，是真事,連金額、天數都是原始數字。

### 三個設計思想

**情境包**：多數辦事指南是「單一業務」的說明（辦護照、辦印鑑證明），但真實世界的麻煩往往是「一件事要跑好幾個單位」。情境包把一串連鎖業務（貸款 → 分區證明 → 權狀補發 → 印鑑證明）串成一條路徑，讓使用者出門前就知道全貌，而不是卡在第三關才發現還有第四關。

**櫃檯會問**：印鑑證明那一趟被「幾份？用途？」問倒是整個專案裡影響最深的一次踩坑。所以每份文件旁邊直接放一個金框「🗣 櫃檯會問」，寫死份數與用途的標準答案。原則是：份數與用途由收件單位決定，去戶政前先問死地政要什麼，不要空手去猜。

**抱怨 → 查證 → 變成後人範例**：站上有個「卡關案例」抱怨區，使用者回報卡關經歷用 mailto 送出，狀態先是「徵解法中」；查到官方依據後補上解法、狀態改成「已解決」，變成後面遇到同樣坑的人可以直接抄的範例。這是整個內容庫的成長機制,不是一次寫死,而是持續由真實回報餵養。

### 零後端的取捨

整站是純前端 SPA（`index.html` + `js/data.js` 內容資料 + `js/app.js` + `css/style.css`），沒有資料庫、沒有登入、沒有伺服器邏輯。使用者的清單、提醒、賓果卡進度都存在 `localStorage`——好處是不收集任何個資、部署只需要 GitHub Pages 這種純靜態代管、沒有維運成本；代價是資料不跨裝置同步、抱怨回報得靠 mailto 開啟郵件 App 而不是真的送進資料庫。這個取捨在專案規模（一人維護、內容型網站、無需帳號體系）下是划算的——先把「零維運成本」換到手,真的需要跨裝置同步再說。

每筆資料標 `lastUpdated`，查證過的加 `verified: "YYYY-MM-DD"` 顯示綠色✓；查不到官方依據的就留著「待查證」，不用記憶硬補。全站掛免責聲明：內容為民間整理,非官方資訊。

## 二、解決問題實錄

### 1. 護照工作天數：查證抓到的錯

網路上流傳護照一般件 4 個工作天就能拿到,這個數字進了初版的 `data.js`。第五輪查證找 3 個 sonnet 子代理逐項比對外交部領事事務局官網,發現現行規定是**一般件 10 個工作天**,4 天是舊制或以訛傳訛的錯誤資訊。這不是小差距——如果使用者照 4 天規劃出國日期,會白白多等快一週還搞不清楚為什麼護照沒下來。修正後這個錯誤本身還被做成一個「破謠言」案例卡,直接標題就是「護照 4 天就拿得到」——網路上很多舊資訊,現在是 10 個工作天,讓查證結果本身變成防雷內容。

**遇到什麼**：初版內容照網路常識填,沒有逐項對照官方公告。
**怎麼判斷**：規費、期限這類數字性資訊風險最高,錯了會直接造成使用者實質損失（跑錯行程、卡在機場）,列為查證第一優先。
**怎麼解**：分工給多個子代理各自負責一組業務類別,逐項附官方來源 URL 寫成查證報告存在 `docs/verification/`,再讓一個獨立代理做「ACCEPT/REJECT」驗收,不是查完就算數。

### 2. 房屋稅退稅年限：法條修法追不上的舊資訊

`data.js` 裡原本寫房屋稅溢繳可以退還的年限是 5 年。查證時發現這是 2021 年 11 月 30 日《稅捐稽徵法》第 28 條修正**前**的舊制——現行條文是一般溢繳「10 年」內可申請,若可歸責於政府機關的錯誤則是「15 年」。5 年這個數字如果沒改,會讓使用者誤以為超過 5 年就沒資格申請更正退稅,直接放棄應得的退稅金額。

**遇到什麼**：法條修正過,但坊間資訊(包含 LLM 訓練資料裡的常識)還停留在舊制。
**怎麼判斷**：只要是「年限」「金額」這類會隨修法變動的數字,一律不能憑記憶填,必須查當前有效版本的法條原文。
**怎麼解**：附法條連結（`law.moj.gov.tw` 稅捐稽徵法第 28 條）,把「5 年」改成「10 年 / 15 年」兩層,並在 pending-claims 留下修法生效日期。

### 3. 死連結:查證時才發現的網址錯誤

遺失物招領那頁原本寫的官網是 `lfp.npa.gov.tw`。查證時直接 `curl` 這個網址——DNS 無法解析,網址根本不存在。正確網域其實是警政署「拾得遺失物管理系統」的 `op2.npa.gov.tw`（curl 回應 HTTP 200 確認過）。同一輪也順便抓到 pitfalls 裡寫「身分證掛失打 1996」是錯的,1996 其實是長照專線,身分證掛失管道另外查無官方依據支持這個說法,直接標記待查證,沒有硬掰一個答案。

**遇到什麼**：內容裡寫死的官方網址,使用者點下去卻是死頁面。
**怎麼判斷**：任何寫進內容的外部連結,不能只看「看起來像官網」,要實際發請求驗證能不能連通。
**怎麼解**：`curl` 直接測試兩個候選網址,一個 DNS 失敗一個 200 OK,證據明確,直接改;不確定的敘述（1996 專線）寧可留白也不沿用查無依據的說法。

### 4. 瀏覽器快取咬到自己

PWA 上線後用 service worker 做離線快取,快取名稱是 `run-once-vN`,資產連結是 `?v=N`。維護時改了 `data.js` 內容,升級了 `index.html` 的版本號,卻忘記同步升 `sw.js` 裡的快取版本字串——結果自己在真機上打開網站,看到的還是上一版內容,被自己的快取咬了。

**遇到什麼**：兩個檔案（`index.html` 的 `?v=N`、`sw.js` 的 `run-once-vN`）要手動同步升版,漏一個就會有使用者看不到更新。
**怎麼判斷**：實測發現「明明改了內容,線上還是舊的」,先懷疑快取而不是懷疑部署失敗——PWA 離線優先架構下這是最常見的坑。
**怎麼解**：寫 `tools/bump.sh` 一鍵同步升級兩處版本號,把「記得手動改兩個地方」這種容易漏的操作變成一個腳本呼叫,往後每次改內容固定跑 `sh tools/bump.sh` 再 commit。

### 5. 剪貼簿三層備援

分享按鈕最早只呼叫 `navigator.clipboard.writeText()`,結果在部分瀏覽器環境（尤其行動裝置、非 HTTPS 或非使用者手勢直接觸發的情境）這個 API 會靜靜懸置,既不成功也不報錯,使用者點了分享,什麼都沒發生。

**遇到什麼**：async clipboard API 不可靠,失敗時沒有明確訊號,使用者會以為按鈕壞了。
**怎麼判斷**：`execCommand("copy")` 雖然是棄用 API,但在使用者手勢觸發的當下同步執行、成功與否立刻知道,反而比非同步的新 API 可靠;async clipboard 則設一個 900ms 逾時,超過還沒回應就視同失敗。
**怎麼解**：三層備援——`execCommand` 同步複製優先 → 失敗改 `navigator.clipboard.writeText()` → 900ms 沒回應或也失敗,彈出一個可以長按全選文字的手動複製面板,保底一定要讓使用者拿到要分享的文字。這套 `copyText()` 後來也是 `shareText()`（原生分享面板）失敗時的共用備援。

### 6. 正則合併踩坑:分類最後一筆會被吞掉

第八輪把 23 項業務內容一次補齊時,是用正則表達式在 `data.js` 裡定位「某分類的最後一筆資料」再插入新內容。踩到的坑是:如果目標剛好是整個分類區塊的最後一筆,「抓到下一筆開頭」當作插入點的正則會多吞一段,把下一個分類的開頭一起吃掉,插入位置就錯了。

**遇到什麼**：用正則表達式定位插入點,在區塊邊界（分類的最後一筆）行為和預期不一致。
**怎麼判斷**：批次修改資料檔案前,插入點如果是靠「找下一段開頭」這種相對定位,一定要對邊界情況（第一筆/最後一筆）額外檢查,不能假設中間值和邊界值行為一致。
**怎麼解**：改用錨點式插入——直接定位到明確、唯一的字串(例如某個 id 的收尾符號)插入,不依賴「下一段在哪裡」這種相對關係。全站 38 筆詳頁改完後逐一渲染做煙霧測試,確認每筆資料都正常顯示才收工。

## 三、應用:同一套方法怎麼套到你自己的主題

這個專案的方法論其實跟公家機關無關,核心是四步:

1. **真實痛點是題目來源,不是腦力激盪**。App 裡分量最重的情境包、影響最深的設計決定（櫃檯會問金框）,全部來自同一次真實踩坑,不是先列 feature 清單再找場景套。如果你想做別的主題的指南工具,先問自己:我(或身邊的人)最近被什麼事卡到崩潰?那件事裡「早知道就好」的每一個瞬間,就是你的內容大綱。

2. **把痛點拆成結構化資料,不是寫成一篇文章**。這個專案沒有把攻略寫成長文,而是拆成 `steps`（時間軸）、`pitfalls`（卡關 Q&A）、`spec`（櫃檯會問標準答案)、`verified`（查證狀態)這些欄位,存在 `data.js` 裡當資料,UI 再統一渲染。好處是內容可以逐筆查證、逐筆更新,不會因為改一個數字就要重寫整段文字。

3. **用子代理分工查證,不要用記憶填答案**。任何會隨時間、修法、政策變動的數字（規費、天數、年限、網址),都不能讓 LLM 憑訓練時的記憶回答——這個專案犯過的每一個錯（護照天數、退稅年限、死連結）都是這種資訊。做法是把查證拆給多個子代理分工、附官方來源連結,查不到就留「待查證」,絕不用記憶硬補一個看起來合理的答案。

4. **小步驗證,線上驗收**。每次改動都是:改資料 → 跑版本升級腳本(避免快取咬自己)→ 瀏覽器裡實際跑一次流程 → commit、push。不是寫一大批功能才一次上線,而是每個情境包、每個修正查證完就推上線,讓真實使用者盡快看到修正。

同一套方法換到你的主題:例如租屋踩坑指南、留學申請卡關實錄、寵物就醫流程——結構都一樣:真實痛點開場,拆成結構化資料,查證別用記憶,小步上線。工具不必是公家機關的樣子,語氣、視覺風格(這個專案用的是「公文紙感」:米白紙底、印泥紅、蓋章式時間軸)完全可以換成貼近你自己主題的樣子。
